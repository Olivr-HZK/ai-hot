# 抓取数据优先本地保存与部分结果续跑方案

## Summary
- 改造 TikTok、X、INS 抓取链路：只要某个平台已经成功抓到数据，就立即写入本地 checkpoint，避免后续额度耗尽、接口失败或进程异常导致已抓数据丢失。
- 当抓取中途失败但本轮已有可用数据时，后续筛选流程继续使用部分数据执行，平台状态标记为 `partial`，不再整个平台直接失败。
- 保持现有最终输出路径兼容：`filtered-result.json`、`raw_posts.json`、`hotspots_*.json` 仍按原逻辑供 `--skip-scrape` 和日常流程使用。

## Key Changes
- 新增统一 checkpoint 目录与配置：
  - 默认目录：`skill_runs/scrape_checkpoints/<platform>/`
  - `.env.example` 新增：
    - `SCRAPE_CHECKPOINT_DIR=skill_runs/scrape_checkpoints`
    - `SCRAPE_PARTIAL_CONTINUE=true`
  - checkpoint JSON 使用原子写入：先写 `.tmp`，再替换正式文件。
  - 每个平台至少写：
    - `latest_raw.json`：最近一次抓到的原始数据快照。
    - `runs/<runId>_raw.json`：本轮归档。
    - `latest_status.json`：记录 `full | partial | failed | skipped`、已完成 query/profile、失败原因、itemCount。

- TikTok 抓取逻辑调整：
  - Apify 每完成一个 keyword，就立刻把当前累计 `rawData` 写入 checkpoint。
  - Apify 额度耗尽时，不再清空已抓到的 Apify 数据。
  - RapidAPI fallback 只抓“当前失败 keyword + 剩余 keyword”的前 10 个，合并到同一批 `rawData`，避免重复抓已经成功的 keyword。
  - RapidAPI fallback 若也失败，但此前已有 Apify 数据，则保存 partial `filtered-result.json` 并返回成功给后续筛选；若没有任何数据才返回失败。
  - 任意 query 级错误发生后，先保存 checkpoint；若已有数据且 `SCRAPE_PARTIAL_CONTINUE=true`，则进入 partial finalize。

- X 与 INS 抓取逻辑同步加固：
  - X RapidAPI 每完成一个搜索词/分页，就保存 raw checkpoint；中途失败时用已抓数据生成 `filtered-result.json`。
  - INS 每完成一个 creator 的帖子抓取，就更新 `skill_runs/instagram/raw_posts.json` 和 checkpoint；后续 creator 失败不影响已有结果。
  - INS 当前“单个 creator 失败跳过”的行为保留，但保存时机提前到每个 creator 成功之后。

- 平台阶段与监控输出：
  - 平台脚本遇到 scraper 非 0 退出时，优先检查本轮 fresh checkpoint / fresh filtered file；存在可用数据则继续筛选，平台结果记为 `partial_success`。
  - `pipeline_monitor/latest.json` 的 `platformResults` 增加可选字段：
    - `scrapeStatus`: `full | partial | failed | skipped`
    - `checkpointPath`
    - `partialReason`
  - 若 partial 数据最终筛选出热点，仍进入合并与飞书阶段；若筛选后 0 条，则按现有 0 条逻辑处理。

## Test Plan
- TikTok Apify 第 11 个 key 跑完部分 keyword 后额度耗尽：确认 checkpoint、`filtered-result.json` 均保留已抓数据，后续筛选继续执行。
- TikTok Apify 耗尽后 RapidAPI 两个 key 也耗尽：已有 Apify 数据不丢失，平台结果为 `partial_success`，不是直接失败。
- TikTok 完全无可用数据且所有 key 失败：平台仍失败，monitor 写明失败原因。
- X 某个搜索词请求失败：此前已抓分页保存，本轮可继续用 partial 数据筛选。
- INS 某个 creator 失败：此前 creator 的帖子已经写入 `raw_posts.json`，不会因后续失败丢失。
- 正常全量运行：最终 raw、filtered、hotspots 输出与原行为兼容，monitor 显示 `scrapeStatus=full`。
- 静态校验：运行 `node --check trend-scrap\tiktok-scraper\src\scraper.js` 和 `node --check trend-scrap\x-scraper\src\scraper.js`，并用 `python -m py_compile` 检查改动的 Python 文件。

## Assumptions
- 用户已确认：中途失败但已有抓取结果时，后续流程应继续处理部分数据。
- 本方案只保证未来运行不丢数据；今天上午已经因内存清空而未落盘的 TikTok Apify 临时结果，不在本次代码修复中追溯恢复。
- checkpoint 是内部恢复与监控文件，不新增飞书字段，不改变飞书写表结构。
